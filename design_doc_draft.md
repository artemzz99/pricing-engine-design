# Design Doc: Personalized Pricing & Offer Engine

> Продуктовая линейка и кросс-продуктовые эффекты — [case.md](case.md).

---

## 1. Контекст

### 1.1 Бизнес-постановка

Крупный розничный банк (~5 млн клиентов, массовый сегмент физлиц) реализует стратегию **максимизации прибыли на длинном горизонте (1–2 года)**. Прибыль формируется на портфеле из четырёх продуктов розничной линейки: дебетовая карта, кредитная карта, депозит/накопительный счёт, премиальный пакет. У каждого продукта есть набор настраиваемых параметров — **тариф** (ставки, лимиты, кэшбэк, условия обслуживания и т. д.).

На текущий момент:

- **Тарифы стандартизированы** — все клиенты получают одинаковые ставки и условия. Дифференциация — только на уровне крупных сегментов (payroll-клиенты, сотрудники банка).
- **Работа с базой ведётся по ручным правилам** — маркетинг определяет сегменты и пресеты офферов; цикл обновления — недели.
- **Управление коммуникациями** (push, sms/email, баннеры в приложении/сайте) опирается на отдельные системы, оптимизирующие краткосрочные метрики Propensity-to-buy и Propensity-to-click.

Подробное описание продуктовой линейки и тарифных параметров — [case.md §2](case.md).

### 1.2 Кросс-продуктовые эффекты

**Ключевая особенность задачи:** изменение тарифа по одному продукту влияет на доходность других продуктов. Решение «по одному продукту независимо» — субоптимально.

| Действие | Прямой эффект | Структурный эффект |
|---|---|---|
| Повышенный кэшбэк на дебетовой карте | ↑ расходы на кэшбэк | ↑ POS-обороты → ↑ interchange · ↑ привычка использовать карту → ↑ остатки → ↑ доход от размещения |
| Активация Премиального пакета | Расходы на привилегии | ↑ остатки ≥ 1,5 млн → ↑ доход от депозитов · ↑ обороты ≥ 150к → ↑ interchange · ↑ NPS → ↓ churn |
| Повышенная ставка по депозиту | ↑ расходы на проценты | ↑ остатки → выполнение условий Премиум → доступ к привилегиям → ↑ обороты |
| Снижение ставки по кредитной карте | ↓ доход от процентов | ↑ активация КК → ↑ POS-обороты → ↑ interchange · ↑ кросс-продажи (доверие к банку → депозит здесь же) |

Полный разбор — [case.md §3](case.md).

### 1.3 Текущий процесс принятия решений по тарифам

- **Источник правды о клиенте** — внутренние банковские данные (продуктовая корзина, транзакции, обороты, остатки, демография, история коммуникаций).
- **Маркетинг** определяет крупные сегменты вручную; присваивает сегменту пресет оффера.
- **Изменение оферты** под персональный тариф — возможно, но требует согласования продуктовой и юридической команд (срок — недели).
- **Цикл планирования** — годовой (бюджет, KPI), что задаёт нижнюю границу горизонта оценки эффекта от изменения тарифа.

## 2. Проблемы

### 2.1 Стандартные тарифы → неоптимальное распределение бюджета

Ставки и промо для всех одинаковые. Клиент с высоким потенциалом получает тот же оффер, что и клиент с низким:

- клиент с высоким потенциалом **недополучает ценности** → выше риск ухода к конкуренту;
- клиент с низким потенциалом **получает избыточную ценность** → расход бюджета без отдачи.

Бюджет на привлечение и удержание распределяется неоптимально, доходность портфеля ниже потенциальной.

### 2.2 Short-term bias в коммуникациях

Текущие системы коммуникаций оптимизируют **краткосрочные** метрики:

- **Propensity-to-buy** — вероятность оформить продукт в ближайшее время;
- **Propensity-to-click** — вероятность кликнуть по баннеру.

Клиент, который при правильной коммуникации принёс бы высокий доход через 6–9 месяцев, отсекается на этапе подбора оффера из-за низкого P-to-buy / P-to-click конкретного продукта в моменте. Долгосрочная ценность клиента не учитывается.

### 2.3 Игнорирование структурных эффектов

Решения по тарифам принимаются по каждому продукту отдельно. Структурное влияние одного продукта на другие (см. §1.2) — не учитывается. В результате:

- комбинации тарифов по продуктам клиента — субоптимальные;
- прямой убыток по одному продукту, который окупается на других, не считается приемлемым в текущей логике.

## 3. Цели и антицели

### 3.1 Цели

1. **Максимизировать прибыль банка на длинном горизонте (1–2 года)** за счёт управления персональными тарифами клиентов, с учётом бюджетных ограничений.
2. **Точно оценивать влияние тарифа** на изменение total-прибыли, получаемой с клиента.
3. **Учитывать внутренние и внешние факторы**, влияющие на total-прибыль с клиента по всем продуктам.

### 3.2 Антицели (что система не делает)

1. **Не используется для KPI / постановки целей отдельных команд** — прогнозы системы не транслируются в индивидуальные KPI продуктовых команд (это создаёт неправильные стимулы).
2. **Не прогнозирует прибыль от продуктов вне розничной линейки** — зарплатные проекты, бизнес-карты и другие B2B-продукты не входят в scope; их финансирование не учитывается в планировании бюджета розницы.
3. **Не прогнозирует макроэкономические показатели** — этим занимается отдельное подразделение, которое поставляет прогнозы по макро-данным как вход.
4. **Не прогнозирует риск-издержки (default, банкротство, фрод)** — этим занимаются отдельные риск-подразделения; их прогнозы используются как вход.

## 4. Требования

### 4.1 Функциональные

- Система оценивает **total-прибыль** по основным статьям банка, на которые влияет розничная продуктовая линейка (4 продукта).
- Система оценивает **влияние изменения тарифа** на total-прибыль клиента на длинном горизонте.
- Система даёт **достаточно точные прогнозы на разных категориях клиентов** — как на new-to-bank, так и на давно пользующихся услугами банка.
- Система даёт прогноз с **достаточной точностью для годового бюджетирования** и распределения бюджета по календарному году.
- Система позволяет **планировать коммуникации с клиентом**, оптимизируя total-прибыль, для двух типов сценариев:
  - тарифы по **X-sell** (новый продукт клиенту);
  - тарифы по **действующим продуктам** клиента.
- Система учитывает **ограничения по тарифным сеткам**, выставленные продуктовой командой (диапазоны ставок, лимиты).

### 4.2 Нефункциональные

- **Покрытие:** работа со всей активной клиентской базой (~5 млн), за исключением сотрудников.
- **Свежесть данных:** использует наиболее актуальные данные о клиентах (realtime / daily в зависимости от источника); прогнозы обновляются на основе этих данных в realtime (до 1 сек).
- **Интеграция:** в действующие CRM-системы для рассылок коммуникаций и управления точками продвижения в приложении/сайте.
- **Масштабируемость:** при добавлении нового продукта или нового тарифного параметра.

## 5. Риски

### 5.1 Налоговые риски

При персональных тарифах, не описанных в общей оферте, существует риск, что повышенный кэшбэк или выгодные условия отдельным клиентам будут признаны налоговой как **доход физлица (НДФЛ-облагаемый)**. Это влечёт **35% налог** на сумму выплат клиенту (по аналогии с рекламными выигрышами свыше 4 000 ₽/год).

**Митигация:** учесть в постановке оптимизации как ограничение или штраф; согласовать с юристами рамки персонализации в рамках общей оферты.

### 5.2 Регуляторные риски

- **ЦБ:** возможные требования к раскрытию условий, ограничения по дискриминации.
- **ФАС:** жалобы клиентов на персонализацию могут трактоваться как недобросовестная конкуренция.

**Митигация:** правовая экспертиза диапазонов персонализации; прозрачное информирование клиента о персональном характере оффера.

### 5.3 Сетевые эффекты и клиентское недовольство

Клиенты обмениваются информацией (семья, коллеги). Резкая разница в условиях между клиентами с похожим профилем создаёт риск:

- падение NPS и репутационный ущерб;
- виральные жалобы в соцсетях.

**Митигация:** контроль разброса условий внутри узких клиентских кластеров; явная коммуникация «персональное предложение».

### 5.4 Риск экстремальных (не рыночных) тарифов

Оптимизатор без ограничений может выдавать тарифы вне рыночного диапазона — нерелевантные ставки, нереалистичные условия.

**Митигация:** жёсткие ограничения от продуктовой команды на тарифную сетку; clipping в постановке оптимизации.

### 5.5 ML-риски

- **Drift:** распределение признаков и таргета смещается со временем (макроэкономика, поведение клиентов, продуктовые изменения).
- **Training-serving skew:** расхождение между значениями фичей в обучении и в проде.
- **Treatment leakage:** утечка информации о применённом тарифе в признаки → завышенная оценка эффекта.
- **Uplift bias:** некорректная оценка инкрементального эффекта тарифа из-за пересечения с естественным поведением клиента.

## 6. Метрики

### 6.1 Метрика верхнего уровня (North Star)

**Выбор:** **Gross Income (GI) 12 месяцев** — суммарный доход с клиента по всем продуктам за вычетом прямых расходов, связанных с обслуживанием этих продуктов, на горизонте 12 мес.

Для каждого продукта *i* клиента *c*:

> **GI(c, i) = Выручка(c, i) − Прямые расходы(c, i)**

где Выручка = interchange + проценты по кредиту + маржа по депозиту + подписка (Премиум); Прямые расходы = кэшбэк + проценты по депозиту + себестоимость привилегий + Risk Cost (для кредитных продуктов).

**GI(c) = Σ GI(c, i)** по всем продуктам клиента.

#### Сравнение вариантов NS-метрики

| Метрика | Плюсы | Минусы |
|---|---|---|
| **CLTV** (Customer Lifetime Value) | Управляет стратегией и долгосрочной траекторией жизни клиента | **Не существует как реально измеримая величина** — для замера нужно дождаться конца жизни клиента в банке. Не валидируется на практике |
| **Gross Income 12 мес** ✅ | 12 мес — удобно для планирования бюджета и инвестиций. Учитывает доходы и расходы, **напрямую связанные с деятельностью системы** | Не включает OPEX. Год — не самый длинный срок для стратегических решений |
| **NPAT 12 мес** | Все плюсы GI + учитывает OPEX, точнее планирует бюджет | Не все затраты в NPAT можно напрямую связать с деятельностью системы. Сложнее считать |

**Решение:** GI 12 мес — лучший компромисс между «существует и измеряется» и «связан со стратегией банка». NPAT отложен — переусложнение для MVP; CLTV отвергнут — невалидируем.

### 6.2 Иерархия метрик

Декомпозиция от North Star (GI 12 мес) вниз по уровням:

- **Бизнес-уровень:** GI 12 мес на клиенте; агрегированный GI на портфеле; ΔGI per offer.
- **Online (продуктовый):** uplift в GI от применённого тарифа vs контрольной группы; конверсия в применение тарифа; retention и churn rate сегмента.
- **Offline / proxy:** предсказанный ΔGI на исторических данных; uplift на holdout; calibration предсказаний относительно факта.
- **Loss function:** регрессия на ΔGI residual либо uplift loss (T-learner / X-learner / DR-learner) — выбор обусловлен типом причинной модели.

Связь с продуктовыми KPI отдельных команд — через декомпозицию GI(c, i) на продуктовые компоненты, без переноса прогнозов в индивидуальные KPI (см. антицель §3.2).

### 6.3 ML-метрики

Качество моделей оценивается по двум осям:

- **Точность предсказания дохода:** RMSE / MAE / MAPE на ΔGI; calibration plot на разных квантилях клиентской базы.
- **Качество ранжирования по uplift:** Qini-coefficient, AUUC, top-decile uplift.

Связь с offline-proxy выше: ML-метрика должна быть **сильным предиктором** offline-proxy метрики, иначе цепочка «модель → бизнес» рвётся.

## 7. Данные

### 7.1 Источники

- **Продуктовая корзина клиента** — текущие и закрытые продукты, даты открытия/закрытия, текущие тарифы, история тарифных изменений.
- **Транзакционные данные** — POS-обороты, MCC-категории, средние чеки, сезонность.
- **Балансы и остатки** — динамика остатков на счетах, депозитах, ссудной задолженности.
- **Демография и сегментация** — возраст, регион, payroll-статус, профессиональный сегмент.
- **История коммуникаций** — отправленные офферы, отклики, отказы.
- **Внешние данные** — макроэкономические прогнозы (от отдельного подразделения), ставки ЦБ, инфляция.
- **Risk Cost** — прогнозы дефолтов и потерь по кредитным продуктам (от риск-подразделения).

### 7.2 SLA и свежесть

- **Realtime (до 1 сек):** баланс, POS-операция, статус активных продуктов.
- **Daily:** агрегированные обороты, остатки, изменения сегментации.
- **Monthly:** макропрогнозы, обновление Risk Cost.

### 7.3 Cold-start

- **New-to-bank:** ограниченные транзакционные сигналы; используются демографические признаки + признаки канала привлечения + аналоги по сегменту.
- **Cold-product:** клиент без продукта *i* — фичи строятся на потенциале (доходный профиль, поведенческие сигналы по аналогичным клиентам).

### 7.4 Bias и качество

- **Survivorship bias:** в исторических данных видны только применённые тарифы — для оценки альтернатив нужны контрольные группы или каузальные методы.
- **Treatment leakage:** значения признаков, на которые повлиял применённый тариф, не должны попадать в фичи без явного контроля.
- **Метаданные** на каждой записи: timestamp, версия источника, источник, способ получения (сырьё / расчёт / вход от внешней команды).

## 8. Бейслайн

### 8.1 Иерархия бейслайнов

- **Constant:** все клиенты получают единый средний тариф по продукту. Smoke test для системы — модель должна быть лучше этого по любой метрике.
- **Rule-based (текущий процесс):** действующая логика CRM — сегменты × пресеты офферов. Реальный baseline, против которого считается uplift и ROI системы.
- **Linear:** GLM или логистическая регрессия на продуктовых фичах с базовым взаимодействием тариф × сегмент.
- **Per-product специализированный:** uplift-модель по одному продукту без учёта кросс-эффектов — изолирует вклад «одно-продуктовой» логики.

### 8.2 Роли бейслайнов

Каждый baseline играет одну из трёх ролей:

- **Smoke test** — Constant: убедиться, что пайплайн работает.
- **Сравнение** — Rule-based: показать business-uplift системы.
- **Fallback** — Linear или Rule-based при недоступности основной модели в проде.

## 9. Валидация

### 9.1 Схема

- **Time-series CV с gap.** Финансовые данные имеют сильную временную зависимость; train и test разделены временным окном (не менее 1 месяца) для предотвращения утечки.
- **Hold-out на финальной оценке** — отдельное временное окно, не использованное в подборе гиперпараметров.
- **Group-aware split** — клиент целиком попадает либо в train, либо в test; не допускается утечка через семьи / связанные счета.

### 9.2 Адверсариальная валидация

Проверка того, что распределения train и test близки. Расхождение → пересмотр временных окон, фичей, целевой группы клиентов.

### 9.3 Валидация uplift-моделей

- **Qini-curve** на holdout — основной критерий ранжирования.
- **A/A simulation** на исторических данных — проверка, что система не выдаёт «эффект» там, где его нет.

## 10. Обзор решений на рынке

### 10.1 Банки и финтехи

- **Тинькофф:** персонализированные категории кэшбэка ежемесячно — клиент выбирает из персонального пула; банк оптимизирует пул.
- **Сбербанк:** сегментные пресеты по премиальным продуктам, индивидуальные ставки по депозитам для приоритетных сегментов.
- **Revolut, N26:** динамические лимиты и персонализированные предложения по подпискам.

### 10.2 Смежные индустрии

- **Авиаперевозки (yield management):** динамическое ценообразование с учётом загрузки, спроса, времени до вылета — каноническая задача, давно решённая, но без структурных эффектов между продуктами.
- **E-commerce (Amazon, маркетплейсы):** персонализированные скидки и промо-цены по категориям клиента; uplift-моделирование как стандарт.
- **Телеком:** retention-офферы при риске оттока, adaptive pricing подписок.

### 10.3 Что переиспользуется, что специфично

- Переиспользуется: uplift-моделирование, A/B-инфраструктура, Qini-метрики, контроль распределений.
- Специфично для банка: регуляторные ограничения, налоговые риски (см. §5.1), кросс-продуктовые эффекты как первый-class объект оптимизации, Risk Cost как составляющая GI кредитных продуктов.

## 11. Cost-of-mistake

Денежная оценка цены текущей проблемы — сколько банк теряет на стандартизированных тарифах относительно персонализации.

### 11.1 Прямые потери

- **Недополученный GI** на сегменте с высоким потенциалом, который ушёл к конкуренту из-за невыгодных стандартных условий.
- **Переплата** на сегменте с низким потенциалом — повышенный кэшбэк или промо, не окупающиеся структурными эффектами.

### 11.2 Потери второго порядка

- **Отток** из-за неконкурентных условий в сегментах с высокой ценовой чувствительностью.
- **Упущенные кросс-эффекты** — клиенты с высоким потенциалом по одному продукту, не активированные на смежных, из-за независимых решений по тарифам.
- **Демотивация продуктовых команд** — невозможность тестировать персональные гипотезы в текущей логике.

Без количественной оценки Relevance & Reasons слабая — это блокер для финальной защиты.

---

## 12. Модель

### 12.1 Архитектура

Рассматриваемые варианты:

- **Multi-task learning:** одна модель предсказывает доход по всем продуктам с разделяемым представлением клиента и продукт-специфичными головами. Учитывает кросс-эффекты через shared embeddings.
- **Каскад:** независимые модели по продуктам + надстройка-оркестратор, которая собирает прогнозы и оптимизирует комбинацию тарифов с учётом структурных эффектов.
- **Causal forest / DR-learner на ансамбле продуктов:** оценка инкрементального эффекта тарифа с учётом наблюдаемых конфаундеров.

Выбор — по результатам валидации на бейслайнах и сложности serving.

### 12.2 Таргет

**ΔGI(c, i, τ) = E[GI(c, i) | tariff = τ] − E[GI(c, i) | tariff = τ_baseline]** — инкрементальный доход от применения тарифа τ относительно baseline-тарифа.

С учётом структурных эффектов — суммарный ΔGI(c) по всем продуктам клиента, с агрегацией по комбинации применённых тарифов.

### 12.3 Учёт тарифа как treatment

Тариф — каузальное воздействие, не корреляционный признак. Подходы:

- **T-learner:** отдельные модели на лечёных и контрольных, разница как uplift.
- **X-learner / DR-learner:** более устойчивы при дисбалансе treatment-групп.
- **Causal Forest:** для интерпретируемости и оценки treatment-effect heterogeneity.

## 13. Оптимизация

### 13.1 Постановка

Для клиента *c* выбрать комбинацию тарифов τ = (τ_1, ..., τ_k) по доступным продуктам, максимизирующую:

> **max Σ ΔGI(c, i, τ_i) + S(c, τ)**

где S(c, τ) — структурные эффекты от комбинации (см. §1.2).

Ограничения:

- **Бюджетные:** Σ расходов системы ≤ B (на портфель / сегмент).
- **Тарифные сетки:** τ_i ∈ Θ_i — допустимый диапазон от продуктовой команды.
- **Налоговые:** учёт 35% НДФЛ при выходе за общую оферту.
- **Ограничение разброса:** max_c τ_i − min_c τ_i ≤ Δ внутри узких кластеров клиентов (для сетевых эффектов, см. §5.3).

### 13.2 Алгоритм

- **Перебор по дискретным пресетам** для MVP — k продуктов × m пресетов = k·m комбинаций на клиента, выбор argmax.
- **MILP / Lagrangian relaxation** для бюджетных ограничений на портфеле.
- **Распределённое решение** на 5 млн клиентов с шардированием по сегментам.

## 14. Serving и инфраструктура

### 14.1 Latency и режим работы

- **Realtime (до 1 сек):** для онлайн-каналов (приложение, сайт) — точечный запрос «оптимальный оффер для клиента».
- **Batch (daily):** для CRM-рассылок — пересчёт офферов на всю активную базу за ночь.

### 14.2 Feature store

Единое хранилище фичей с гарантией консистентности train/serve:

- batch-агрегаты обновляются по расписанию;
- realtime-фичи рассчитываются на лету через streaming-pipeline;
- метаданные (версия, timestamp, источник) пишутся вместе со значением.

---

## 15. Evaluation

### 15.1 Offline

- Backtest на историческом периоде с time-series CV.
- Симуляция на synthesised counterfactuals для проверки uplift-моделей.

### 15.2 Online (A/B)

- **Unit of randomization:** клиент.
- **Гипотеза:** ΔGI на test-группе ≥ ΔGI на control-группе с MDE = X% при α = 0.05, β = 0.2.
- **Длительность:** не менее 12 недель для стабилизации эффекта; финальная оценка — на годовом окне.
- **Guardrails:** NPS, churn rate, число жалоб (для §5.3); тарифные диапазоны в пределах сеток (для §5.4).

---

## 16. Мониторинг

Пирамида сверху вниз:

- **Бизнес-KPI:** GI 12 мес, ΔGI vs контрольная группа.
- **Качество модели:** uplift на свежих данных, calibration drift.
- **Качество данных:** PSI / Wasserstein на ключевых фичах (порог: PSI > 0.2 → alert).
- **SW health:** latency, error rate, доступность feature store.

Training-serving skew — отдельный alarm: расхождение распределений фичей в обучении и проде.

---

## 17. План реализации

### 17.1 MVP scope

- 1 продукт (дебетовая карта) × 3–5 пресетов тарифа.
- Rule-based + Linear baseline + одна uplift-модель (T-learner).
- Offline-валидация + pilot A/B на ограниченном сегменте.

### 17.2 Этапы

1. **Discovery:** данные, бейслайн, baseline-метрики.
2. **MVP-модель:** T-learner на 1 продукте, offline-валидация.
3. **Pilot A/B:** 2–4 недели на 5–10% базы.
4. **Расширение:** добавление продуктов 2–4, multi-task / каскад.
5. **Оптимизатор:** комбинации тарифов с учётом структурных эффектов и бюджетных ограничений.
6. **Production:** realtime-serving, feature store, мониторинг.

## Глоссарий

- **Тариф:** набор параметров продукта, конфигурируемых для клиента (ставка, кэшбэк, лимит, срок, комиссии).
- **Пресет:** фиксированная комбинация значений параметров тарифа, согласованная продуктовой командой.
- **Interchange:** комиссия, которую банк-эмитент получает от платёжной системы при безналичной покупке клиента.
- **Risk Cost:** ожидаемые потери по кредитному продукту из-за дефолтов.
- **POS-обороты:** объём безналичных покупок клиента за период.
- **GI (Gross Income):** доход по продукту = выручка − прямые расходы.
- **NPAT (Net Profit After Tax):** чистая прибыль по продукту после налогов.
- **NS-метрика (North Star):** ключевая метрика верхнего уровня, на оптимизацию которой направлена система.
- **X-sell (cross-sell):** предложение нового продукта существующему клиенту банка.
- **New-to-bank:** клиент без истории в банке.
- **Uplift:** инкрементальный эффект от применения treatment относительно контрольной группы.
- **Qini / AUUC:** метрики качества uplift-модели.
- **PSI (Population Stability Index):** мера сдвига распределения фичи во времени.
